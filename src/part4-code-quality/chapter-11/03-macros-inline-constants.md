# 11.3 Macros, Inline Functions, and Constants

## 1. The Preprocessor: A Text-Substitution Engine
The C preprocessor is an incredibly powerful tool that has historically been abused by embedded engineers to compensate for the limitations of 1980s compilers and 8-bit microcontrollers. However, the preprocessor operates entirely outside the language rules of C. It is a blind text-substitution engine that runs *before* the compiler even sees the code. It has no concept of data types, variable scope, or memory safety.

Using `#define` for constants or function-like macros removes all compiler guardrails, breaks debuggers (which cannot "step into" a macro because it doesn't exist at runtime), and introduces catastrophic, invisible bugs. In modern firmware architecture, we must violently reject the preprocessor wherever a type-safe alternative exists.

### 1.1 The Anti-Pattern: Function-Like Macros and Side Effects
Decades ago, calling a function on an 8051 or a PIC microcontroller was expensive. It required pushing registers onto the stack, executing a branch instruction, and popping the stack on return. To save CPU cycles, developers used function-like macros to inline the logic.

The most infamous example is the `MAX` macro.

```c
/* ANTI-PATTERN: The perilous function-like macro */
#include <stdint.h>
#include <stdio.h>

// 1. We define a simple macro to return the maximum of two values.
// We even remember to wrap the arguments in parentheses to prevent 
// operator precedence bugs (a classic preprocessor trap).
#define MAX(a, b) (((a) > (b)) ? (a) : (b))

void process_sensor_data(void) {
    uint8_t sensor_reading = 10;
    uint8_t threshold = 5;
    
    // 2. The developer decides to increment the sensor reading while checking it.
    // This is a "side effect" (a state change).
    uint8_t highest_val = MAX(sensor_reading++, threshold);
    
    // 3. What is the value of sensor_reading? It should be 11.
    // But what did the preprocessor actually generate?
}
```

**The Deep Technical Rationale (The Double Evaluation Bug):**
The preprocessor blindly substitutes the text `sensor_reading++` every time it sees the argument `a`. The code handed to the compiler looks like this:

`uint8_t highest_val = (((sensor_reading++) > (threshold)) ? (sensor_reading++) : (threshold));`

Because `10 > 5` is true, the first `sensor_reading++` evaluates to 10 (and increments the variable to 11). Then, the ternary operator executes the true branch: `(sensor_reading++)`. This evaluates to 11, and increments the variable *again* to 12. 
The developer intended to increment the sensor reading once. The macro silently incremented it twice. This bug can corrupt an entire state machine and is nearly impossible to find during a code review because the code *looks* perfectly correct.

### 1.2 The Production-Grade Solution: `static inline`
Modern compilers (GCC, Clang, ARM Compiler 6) possess optimizers that are vastly superior to human hand-tuning. We no longer need to rely on the preprocessor for performance. 

Instead, we use `static inline` functions. 

```c
/* PRODUCTION-GRADE: Type-safe inline functions */
#include <stdint.h>

// 'static' gives the function internal linkage (visible only to this file).
// 'inline' suggests to the compiler that the function body should be inserted 
// directly into the caller's assembly, completely eliminating the stack/branch overhead.
static inline uint8_t max_u8(uint8_t a, uint8_t b) {
    return (a > b) ? a : b;
}

void process_sensor_data(void) {
    uint8_t sensor_reading = 10;
    uint8_t threshold = 5;
    
    // The compiler enforces types and executes side-effects exactly once!
    uint8_t highest_val = max_u8(sensor_reading++, threshold);
    
    // sensor_reading is correctly 11. The generated assembly is identical 
    // to the macro, but 100% type-safe.
}
```

The `static inline` function provides the best of both worlds: the performance of a macro and the strict type-checking, scope rules, and side-effect safety of a real C function.

### 1.3 The Anti-Pattern: `#define` for Constants
Using `#define` to create system constants is a legacy habit that destroys observability.

```c
/* ANTI-PATTERN: Untyped Preprocessor Constants */
#define MAX_RETRY_COUNT 5
#define SENSOR_I2C_ADDR 0x4A
```

**The Observability Failure:**
When you step through the code in a JTAG debugger (e.g., Segger Ozone or GDB), the debugger only knows about the ELF file generated by the compiler. Because the preprocessor stripped `MAX_RETRY_COUNT` out of the text before compilation, the symbol `MAX_RETRY_COUNT` does not exist in the final binary. If you try to add it to a watch window, the debugger will report "Symbol not found."

Furthermore, `#define` constants have no type. Is `0x4A` a signed 8-bit integer? An unsigned 32-bit integer? The compiler has to guess based on context, which can lead to implicit conversion warnings or bugs (especially on 16-bit architectures where integer promotion rules are complex).

### 1.4 The Production-Grade Solution: `const` and `enum`
For true constants, we use the `const` keyword or `enum` types.

```c
/* PRODUCTION-GRADE: Typed constants and Enumerations */
#include <stdint.h>

// 1. Typed constants. The compiler knows exactly what this is.
// The linker can place it in Flash (ROM) memory, saving SRAM.
// The JTAG debugger can see the symbol 'max_retry_count' and its value.
static const uint8_t max_retry_count = 5U; 

// 2. Enumerations for related sets of constants.
typedef enum {
    I2C_ADDR_TEMP_SENSOR = 0x4A,
    I2C_ADDR_EEPROM      = 0x50,
    I2C_ADDR_RTC         = 0x68
} i2c_device_address_t;

void init_sensor(void) {
    // The compiler enforces that we are passing a valid address type.
    i2c_init(I2C_ADDR_TEMP_SENSOR); 
}
```

### 1.5 The Exception: Multi-Statement Macros and `do { ... } while(0)`
There is exactly one scenario where function-like macros are still architecturally necessary: when you need to capture compiler metadata like `__FILE__` and `__LINE__` for assertions or logging. An inline function cannot capture the caller's line number; a macro can.

If you must write a macro that executes multiple C statements, you **must** wrap it in a `do { ... } while(0)` block.

```c
/* ANTI-PATTERN: The dangling else trap */
#define LOG_ERROR(msg) \
    uart_print("ERROR: "); \
    uart_print(msg)

void check_status(bool failed) {
    if (failed)
        LOG_ERROR("System failure"); // This expands into a disaster!
}
```
If you expand the macro, the `if` statement only applies to the first `uart_print`. The second `uart_print` will execute unconditionally, every single time, regardless of the `failed` variable.

```c
/* PRODUCTION-GRADE: The do-while-zero wrapper */
#define LOG_ERROR(msg) \
    do { \
        uart_print("ERROR: "); \
        uart_print(msg); \
    } while(0) // This creates a single logical block that swallows the semicolon safely.
```

## 2. Company Standard Rules

**Rule 11.3.1:** The creation of new function-like macros (macros that accept arguments) is strictly forbidden for mathematical or logical operations. `static inline` functions MUST be used instead to guarantee type safety and prevent side-effect duplication.
**Rule 11.3.2:** Function-like macros are ONLY permitted when capturing compiler metadata (`__FILE__`, `__LINE__`, `__func__`) is required for logging, assertions, or specialized debugging infrastructure.
**Rule 11.3.3:** If a macro executes multiple statements, it MUST be wrapped in a `do { ... } while(0)` block to ensure it behaves as a single logical statement when placed inside un-braced `if/else` control structures.
**Rule 11.3.4:** The `#define` directive MUST NOT be used for defining integer or floating-point constants. `static const` variables or `enum` types MUST be used instead to provide strong typing and symbol visibility in JTAG debuggers.
**Rule 11.3.5:** All macros MUST be named using `UPPER_SNAKE_CASE`. No function or variable shall use uppercase, ensuring developers instantly recognize preprocessor text-substitution.
